# 反 AI Slop 完整清单

> 来源：huashu-design 反 AI slop 规则，适配 spec-designer 场景。
> 核心原则：**品牌 spec 里明写了才能用，否则默认禁用。**

---

## 什么是 AI Slop？

**AI Slop = AI 训练语料里最常见的"视觉最大公约数"。**

紫渐变、emoji 图标、圆角卡片+左 border accent、SVG 画人脸——这些东西之所以是 slop，不是因为它们本身丑，而是因为**它们是 AI 默认模式下的产物，不携带任何品牌信息**。

**规避 slop 的逻辑链**：
1. 用户请你做设计，是要**他的品牌被认出来**
2. AI 默认产出 = 训练语料的平均 = 所有品牌混合 = **没有任何品牌被认出来**
3. 所以 AI 默认产出 = 帮用户把品牌稀释成"又一个 AI 做的页面"
4. 反 slop 不是审美洁癖，是**替用户保护品牌识别度**

---

## 核心要规避的清单

| 元素 | 为什么是 slop | 破例条件 |
|------|-------------|---------|
| **激进紫渐变** | AI 训练语料里"科技感"的万能公式，出现在每一个落地页 | 品牌 spec 明确写了用紫渐变 |
| **Emoji 作图标** | 训练语料里每个 bullet 都配 emoji，是"不够专业就用 emoji 凑"的病 | 品牌 spec 写了（如 Notion 的 ✅ ✅ ✅ 风格） |
| **圆角卡片 + 左彩色 border accent** | 2020-2024 Material/Tailwind 时期的烂大街组合 | 用户明确要求，且品牌 spec 保留此元素 |
| **SVG 画人/物** | AI 画的 SVG 人物永远五官错位，比例诡异 | **几乎没有**——有图就用真图，没图就留诚实 placeholder |
| **CSS 剪影/SVG 手画代替真实产品图** | 生成的是「通用科技动画」——黑底+橙 accent+圆角长条，任何实体产品都长一样 | **几乎没有**——先走核心资产协议找真图，实在不行用 AI 生成 |
| **Inter/Roboto/Arial/system fonts 做 display** | 太常见，读者看不出这是"有设计的产品"还是"demo 页" | 品牌 spec 明确用这些字体 |
| **赛博霓虹（深蓝底 #0D1117 + 霓虹发光）** | GitHub dark mode 美学的烂大街复制 | 开发者工具产品且品牌 spec 走了这个方向 |
| **编造 stats/quotes 装饰** | 没用的数字、图标装饰充数，显得不专业 | **永远禁止**——留空白或问用户要真实数据 |
| **每个标题都配无意义的 icon** | iconography slop，每个标题都配 icon | 只保留承载差异化信息的密度元素 |

## 正向做什么

| 动作 | 为什么 |
|------|--------|
| `text-wrap: pretty` + CSS Grid + 高级 CSS | 排版细节是 AI 分不清的"品味税"，会用这些的 agent 看起来像真设计师 |
| 用 `oklch()` 或 spec 里已有的色 | 不凭空发明新颜色——所有临场发明的色都会让品牌识别度下降 |
| 配图用 AI 生成（Gemini/Flash）或 Wikimedia/Unsplash | AI 生成的图比 SVG 手画准确，比 HTML 截图有质感 |
| 文案用「」引号不用 "" | 中文排印规范，也是"有审校过"的细节信号 |
| 一个细节做到 120%，其他做到 80% | 品味 = 在合适的地方足够精致，不是均匀用力 |

## 交付前自检流程

在交付 HTML 原型之前，逐项扫描：

```markdown
## 反 AI Slop 自检

- [ ] 没有激进紫渐变（品牌 spec 允许除外）
- [ ] 没有 emoji 图标（品牌 spec 允许除外）
- [ ] 没有 SVG 手画人/物（用真图或 placeholder）
- [ ] 没有 CSS 剪影代替真实产品图（用真图或 placeholder）
- [ ] 没有系统字体做 display（品牌 spec 允许除外）
- [ ] 没有编造 stats/quotes（用真实数据或空白）
- [ ] 没有无意义的 icon 每处配（只保留承载信息的）
```

如果发现命中：
- **永远禁止**的条目 → 必须修复（替换/删除）
- **品牌 spec 允许**的条目 → 确认 brand-spec.md 里有对应声明

## 设计原则 #0 · 事实验证先于假设

> 这条优先级高于反 AI slop——如果事实错了，反 slop 也是在错误基础上做判断。

**任何涉及具体产品/技术/事件的事实性断言，第一步必须 `WebSearch` 验证，禁止凭训练语料做断言。**

触发条件：
- 用户提到具体产品名（如"大疆 Pocket 4"）
- 涉及 2024 年及之后的发布时间线、版本号
- 内心冒出"我记得好像是..."、"应该还没发布"

硬流程：
1. `WebSearch` 产品名 + "2026 latest" / "launch date" / "specs"
2. 读 1-3 条权威结果确认存在性和状态
3. 把事实写进 `product-facts.md`
4. 搜不到 → 问用户，不是自行假设
