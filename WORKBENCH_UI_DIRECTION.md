# Workbench UI Direction

本文档记录当前工作台 UI 的后续演进方向，作为后续改动、spec 编写和原型对齐时的参考依据。

## 核心判断

VibeX Workbench 不应只是展示 spec 文件树，而应帮助用户沿着 spec 图谱完成澄清、推导、确认、实现的闭环。

后续每个 UI 改动都应优先回答一个问题：

> 它是否让用户更快知道当前 spec 缺什么、下一步该做什么？

## 产品方向

当前 UI 已经开始形成“spec 驱动 IDE”的形态，但仍偏向把信息全部摆出来。下一阶段不宜继续增加面板和视觉元素，而应做减法：

- 默认只展示当前决策路径。
- 非当前路径的信息应折叠为可展开上下文。
- 左侧 spec 列表、中央图谱、右侧 Agent Context 应各自承担清晰职责，避免重复表达同一件事。

## 区域职责

### 左侧列表：导航，不做第二个图谱

左侧 Spec Explorer 的职责是快速定位和筛选，而不是完整表达 spec 关系。

应保留：

- 层级
- 标题
- 状态
- 一行摘要
- 搜索和过滤

应弱化：

- 文件名
- 过长路径
- 复杂关系表达

真正的上下游、父子、同级关系应交给中央图谱展示。

### 中央图谱：主工作区

中央图谱应成为用户理解和编辑 spec 关系的主要工作区。

推荐默认聚焦：

- 父级 spec
- 当前 spec
- 直接子级
- 同级候选
- 可添加的下一层节点

不建议默认展示过多全局节点。全局视图可以作为切换模式存在，但主视图应服务于“当前 spec 的上下游决策”。

### 右侧 Agent 面板：操作台，不只是聊天

右侧 Agent 区应逐步从聊天输入框演进为 spec 操作台。

Context 区是正确方向。下一步应把命令能力做成半结构化动作：

- `/add`：添加子节点
- `/validate`：校验现有输入、输出和 spec 是否合理
- `/deduce`：根据当前 spec 和子 spec 推导新子 spec
- `/analyse`：分析当前 spec
- `/confirm`：确认 spec
- `/implement`：实现
- `/implement-loop`：循环直至实现完成

用户点击卡片后，右侧应根据当前层级和状态推荐最合理的 2-3 个动作，而不是要求用户记住所有命令。

## Spec 卡片展示规范

每张 spec 卡片应优先展示用户能理解的业务信息，而不是机器文件信息。

推荐统一展示：

- 精简标题
- 一行摘要
- 层级
- 状态
- 最近验证结果

文件名、相对路径、spec.name 等机器信息应弱化显示，仅作为辅助识别信息。

为支持这一点，spec 中应逐步补齐：

```yaml
display:
  title: "简短中文标题"
  summary: "一句话说明这个 spec 解决什么问题"
  description: "必要时补充更完整的上下文"
```

默认情况下，`display` 内容应由 agent 在创建或修改 spec 时同步生成或更新。

## 视觉方向

主题应继续收敛，而不是为每个区域设计一套独立视觉语言。

建议：

- 层级可以使用固定颜色区分。
- 选中态、hover、告警、成功、失败等交互状态应全局统一。
- 高亮色数量应控制，避免每个面板都有自己的强调色。
- 滚动条、边框、背景层级应贴近 Cursor / VS Code 的原生工作台气质。

## 后续改动原则

1. 优先减少认知负担，而不是增加信息密度。
2. 优先强化当前 spec 的下一步动作，而不是展示完整世界。
3. 左侧负责找，中央负责看关系，右侧负责让 agent 行动。
4. 用户可见信息优先使用中文标题和摘要，机器路径作为次要信息。
5. 任何新增 UI 都应说明它属于导航、关系理解、上下文管理还是执行操作。
